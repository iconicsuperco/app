# Muse — Implementation Plan

A premium, fully-local, installable music web app. **Zero paid dependencies**, 100% client-side, works offline once loaded. Your chosen stack: React + TypeScript, IndexedDB persistence, PWA, plus playlists, lyrics, audio visualizer, and EQ/effects.

---

## 1. Technology Stack (all free, open-source)

| Concern | Choice | Why |
|---|---|---|
| **Framework** | React 18 + TypeScript | Your choice. Best ecosystem for media-rich, component-heavy UIs. |
| **Build tool** | Vite 6 | Instant HMR, first-class PWA plugin, ESM-native. |
| **Language** | TypeScript (strict) | Type safety across the audio engine, DB schema, and player state. |
| **Styling** | Tailwind CSS v4 + CSS custom properties | Utility speed + design tokens for theming. |
| **Component primitives** | Radix UI (headless) + custom shadcn-style components | Accessible (keyboard, ARIA), unstyled so we control the premium look. |
| **Animation** | Framer Motion (motion) | Spring physics, layout animations, gesture support — essential for the fluid feel. |
| **Icons** | lucide-react | Clean, consistent, tree-shakeable. |
| **Routing** | React Router v7 | Mature, type-safe routes, data loaders for album/artist pages. |
| **State** | Zustand | Lightweight global stores for player/queue/library without boilerplate. |
| **Persistence** | Dexie.js (IndexedDB wrapper) | Transactions, indexes, reactive live queries, multi-GB Blob storage. |
| **Audio core** | Web Audio API (custom engine) | Required for EQ, visualizer (AnalyserNode), crossfade, gapless. HTMLAudio alone can't do this. |
| **Metadata** | `music-metadata` (browser build) | Reads ID3/MP4/FLAC/OGG/OPUS tags — title, artist, album, art, genre, year, track#, BPM. |
| **PWA** | `vite-plugin-pwa` (Workbox) | Service worker, web manifest, installable, offline caching of the app shell. |
| **Quality** | ESLint + Prettier + Vitest | Linting, formatting, and unit tests for the audio/queue/DB logic. |

**Why no backend?** Everything is local — no streaming, no accounts, no sync. IndexedDB persists both metadata and the raw audio Blobs. The app loads once and works offline. This is the simplest, most robust architecture for a personal app and exactly matches your "drag files into the browser" vision.

```
│   ├── router.tsx

│   │   ├── importFiles.ts   # drag-drop / file-picker / folder entrypoint
│   │   ├── extractMetadata.ts
│   │   ├── guessFromFilename.ts  # metadata fallback heuristic
│   │   └── extractPalette.ts     # dominant colors from album art
│   ├── queries.ts           # reactive Dexie queries (live queries → Zustand)
│   └── backup.ts            # export/import library + settings (JSON)
│
├── player/                  # Playback state machine
│   ├── PlayerStore.ts       # Zustand: current track, position, status
│   ├── QueueStore.ts        # order, shuffle, repeat modes, history
│   └── shortcuts.ts         # keyboard shortcut registry
│
├── store/                   # UI state (Zustand stores)
│   ├── uiStore.ts           # theme, mini/full player, panels, toasts
│   ├── searchStore.ts
│   └── settingsStore.ts     # volume, EQ prefs, crossfade duration
│
├── features/                # UI grouped by feature (each self-contained)
│   ├── home/  library/  albums/  artists/  songs/
│   ├── recently-added/  recently-played/  favorites/
│   ├── search/  queue/  playlist/  now-playing/  lyrics/
│   ├── visualizer/  equalizer/  settings/
│   └── import/              # "Add Music" drag-drop zone + progress UI
│
├── components/              # Shared, reusable UI
│   ├── ui/                  # Button, Card, Slider, Dialog, etc. (shadcn-style)
│   ├── player/              # MiniPlayer, FullPlayer, ProgressBar, VolumeControl
│   ├── track/               # TrackRow, TrackList (virtualized), TrackMenu
│   ├── layout/              # Sidebar, TopBar, GlassBar
│   └── viz/                 # Canvas-based visualizer renders
│
├── hooks/                   # Reusable hooks (usePlayer, useQueue, useMediaSession…)
├── lib/                     # utils, formatting, color math, id generation
├── styles/                  # global CSS, design tokens, fonts
└── types/                   # shared types
```

**Key design principles:**
- **Separation of concerns:** the `audio/` engine is pure TypeScript with zero React dependencies → testable, swappable. UI subscribes via hooks.
- **Single source of truth:** Dexie is the database; Zustand holds player/queue/UI state. Dexie `liveQuery` feeds derived collections (albums, artists) so the UI auto-updates on import/edit.
- **Modular features:** each feature folder owns its routes, components, and hooks. New features (e.g. smart playlists later) drop in without touching others.

## 3. Data Model (core entities)

- **Track** — id, blobId (audio), artworkId, title, artist, albumArtist, album, genre, year, trackNo, discNo, duration, format, bitrate, sampleRate, lyrics, palette, isFavorite, dateAdded, lastPlayed, playCount
- **Album** — id, title, albumArtist, year, artworkId, palette, trackIds
- **Artist** — id, name, albumIds, trackCount
- **Playlist** — id, name, description, artworkId, trackIds[], createdAt, updatedAt
- **PlayHistory** — id, trackId, playedAt
- **Settings** — volume, eqBands, eqPreset, crossfadeMs, gapless, theme, accentColor, repeatMode, shuffle

Artwork and audio are stored as **Blobs** in IndexedDB (referenced by id), not base64 — efficient for large libraries.

## 4. The Audio Engine (the technical heart)

Custom Web Audio graph per playback session:
```
MediaElementSource → [10× BiquadFilter EQ] → AnalyserNode → PreAmp Gain → Crossfade Gain → Destination
```
- **Gapless playback:** preload next track's AudioBuffer; swap at track end.
- **Crossfade:** two-source fade between current/next for the configured duration.
- **Visualizer:** AnalyserNode FFT data streamed to a Canvas render at 60fps via `requestAnimationFrame` (several visual modes: bars, radial, waveform).
- **Equalizer:** 10-band graphic EQ with presets (Flat, Bass Boost, Vocal, Electronic…) + custom.
- **ReplayGain-style normalization** (loudness leveling) as a quality extra.
- **Media Session API** integration → hardware play/pause keys, lock-screen art.

## 5. Import Pipeline ("Add Music")

1. **Ingest** — drag-drop zone, file picker (multi-select), and folder upload (`webkitdirectory`) all feed one pipeline.
2. **Extract metadata** via `music-metadata` → title/artist/album/art/genre/year/track#/duration.
3. **Fallback** — if tags are missing, parse the filename (`Artist - Title.ext`, `Title`, track-number prefixes) with a smart heuristic.
4. **Artwork & palette** — extract embedded art as a Blob; compute a dominant-color palette for dynamic theming.
5. **Persist** — store audio + art Blobs and a Track record in Dexie; auto-group into Albums & Artists.
6. **Progress UI** — toast/panel showing per-file progress, with skip-on-error resilience.
7. **Library auto-updates** via Dexie liveQuery — songs appear instantly.

## 7. Design System (premium feel)

- **Dark-first** theme on a neutral gray ramp with a signature **Muse gradient** accent (configurable). Album art can drive dynamic, per-screen theming via extracted palettes.
- **Typography:** Inter (UI) + a refined display face for headlines; tight tracking, generous spacing, 4px grid.
- **Glassmorphism:** frosted sidebar + translucent now-playing bar with backdrop blur (used where it aids hierarchy, not everywhere).
- **Motion:** spring-based route transitions, shared-layout animations (tap album → hero morphs), hover lifts, long-press context menus, smooth queue reordering via Framer Motion `Reorder`.
- **Micro-interactions:** heart-fill animation, equalizer "now playing" bars on the active track row, ripple/hover states, skeleton loaders.
- **Accessibility:** Radix primitives (keyboard + ARIA), focus-visible rings, `prefers-reduced-motion` support, sufficient contrast, ARIA live regions for toasts.
